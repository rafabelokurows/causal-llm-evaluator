# Causal LLM Evaluator

Every ML team argues about prompts. Almost none of them can prove who's right.

This evaluator borrows the randomised trial design from clinical medicine and applies it to LLM prompt engineering — so instead of "this one scored higher on my laptop," you get a p-value, a confidence interval, and an effect size that tells you whether the difference is real and whether it's worth shipping.

---

## The Problem

Prompt engineering decisions are usually made by intuition. Someone rewrites a system prompt, eyeballs ten outputs, and ships. This is how teams end up arguing in Slack about which version "feels better" while neither side has evidence.

The specific failure modes:

**Vibe-based comparison.** You run prompt A and prompt B on the same handful of test cases and compare outputs manually. This fails because (a) humans are inconsistent judges, (b) you unconsciously cherry-pick the examples that support your preference, and (c) you have no way to know whether the difference you see would hold on the next 1,000 inputs.

**Average scores on a fixed benchmark.** You score 20 outputs, take the mean, and declare a winner. This fails because the sample is too small to distinguish a real effect from noise. A difference of +0.08 on a 0–1 scale sounds meaningful — but with 20 samples and typical LLM output variance, it's almost certainly within sampling error. You'd need to know the confidence interval to say anything defensible.

**Confounded assignment.** You evaluate prompt A on one set of inputs and prompt B on another. Any score difference could come from one set being systematically easier — not from the prompt itself. Without random assignment, you can't separate the prompt's effect from the input's effect.

**No effect size.** Even teams that do basic significance testing often stop at the p-value. A result can be statistically significant (real) but practically negligible (not worth shipping). You need both.

---

## The Solution

This evaluator treats prompt A/B testing as a proper randomised controlled trial, the same design used in clinical medicine to measure whether a drug actually works.

**Random assignment** — each `(input, sample)` pair is independently assigned to a prompt variant at runtime. This breaks any correlation between the variant and input difficulty. Harder inputs are equally likely to land in any variant.

**Balanced allocation** — each variant receives exactly the same number of trials per input, so effect estimates aren't distorted by unequal group sizes.

**Effect estimation pipeline:**

```
Random assignment (balanced per input)
    → LLM calls (asyncio.gather — all variants run concurrently)
    → LLM-as-judge scoring (0–1 normalised)
    → Average Treatment Effect: ATE = mean(treatment) − mean(control)
    → Bootstrap 95% CI (2000 resamples, no normality assumption)
    → Welch's t-test (p-value — no equal-variance assumption)
    → Cohen's d (practical effect size, independent of sample size)
```

**What you get:**

- **p-value** — is the difference statistically real, or sampling noise?
- **Cohen's d** — is the effect practically meaningful regardless of sample size?
- **95% bootstrap CI** — what's the plausible range of true effects, given observed variance?

You want `significant = true` *and* `Cohen's d > 0.5` before shipping a prompt change. Both conditions together mean the effect is real and worth the change.

## Origin: Primetag Creator Analysis

This tool was built at [Primetag](https://primetag.net) to systematically evaluate prompt variants for creator intelligence reports.

Primetag analyses influencer creators for brand partnership decisions — scoring audience fit, content quality, brand safety, and growth signal. These reports are generated by LLM pipelines, and the quality of the report depends heavily on how the prompt is written.

The specific problem: four different prompt strategies for creator analysis were under consideration — a minimal baseline, a structured rubric, a chain-of-thought approach, and a luxury brand buyer persona. Each "felt" different when read. None had been measured.

Running this evaluator against those four variants with real creator profiles produces a forest plot of Average Treatment Effects with 95% confidence intervals — a defensible, reproducible answer to which strategy actually performs better, by how much, and whether the difference is real.

The `demo/creator_experiment.json` file encodes exactly this experiment. The profiles in it are synthetic but structurally representative of real Primetag creator data: one strong-signal creator (high engagement, fast growth, clean safety record), one micro-creator edge case (strong retention, no brand deals, uncertain scale), and one stress-test case (declining growth plus a past controversy).

**Sample result (experiment `682f9075`, 10 reps/variant, temp 0.2):**

| Variant | ATE vs control | Cohen's d | p-value | Significant |
|---|---|---|---|---|
| `persona_talent_buyer` | +0.033 | 1.00 | 0.175 | No |

No significant differences at 10 reps — expected. `persona_talent_buyer` shows Cohen's d = 1.0 (large effect) with p = 0.175: the effect is likely real but the experiment is underpowered. Rerun with `n_reps_per_variant: 50` to cross the significance threshold. This is exactly the kind of signal that vibe-based comparison would miss or misread as "clearly better."

Each experiment is saved to `experiments.jsonl` — one JSON line per run, with timestamp, all effect estimates, and the interpretation.

## FAQ

**Why use causal inference instead of just comparing averages?**

Averages tell you what happened in your sample. Causal inference tells you whether the difference would hold in general. If you run prompt A on 10 inputs and prompt B on 10 different inputs, any difference in scores could be because A got easier inputs — not because A is actually better. Random assignment eliminates that confound, and the bootstrap CI quantifies how uncertain you should be about the result.

**What's the difference between ATE, p-value, and Cohen's d — which one should I look at?**

All three together. ATE is the raw score difference (e.g. +0.08 on a 0–1 scale). The p-value tells you whether that difference is likely to be noise — below 0.05 means it's probably real. Cohen's d tells you whether it's practically meaningful regardless of sample size — a result can be statistically significant but so small it doesn't matter. You want `significant = true` *and* Cohen's d > 0.5 before shipping a prompt change.

**How is LLM-as-judge scoring reliable if the judge itself is an LLM?**

The judge isn't rating outputs in isolation — it's applying a fixed rubric consistently across all variants. Because the same judge model scores every output in the same experiment, any systematic bias cancels out when you compute the ATE. The risk is judge inconsistency (high variance), which shows up as wide confidence intervals rather than false positives. Wide CIs are the correct signal: they tell you to run more samples, not that the result is wrong.

**Why Welch's t-test instead of Student's t-test?**

Student's t-test assumes both groups have equal variance. Prompt variants that produce more structured, predictable outputs will have lower score variance than open-ended variants — equal variance is almost never a safe assumption here. Welch's t-test handles unequal variances correctly with no real downside.

**Can this compare models (e.g. Claude vs Llama) instead of just prompts?**

Yes. Set the same prompt as both variants but different `provider`/`model` values per variant. The experiment runner handles mixed providers — one variant calls Anthropic, another calls Groq — and the causal estimation runs identically on the scores regardless of source.

**Why bootstrap CI instead of a parametric confidence interval?**

LLM judge scores are bounded (0–1) and often non-normal — parametric CIs assume normality. Bootstrap makes no distributional assumption: it resamples the actual observed scores 2000 times and reads the interval directly off the empirical distribution. More conservative, more honest.

**What's the minimum number of samples needed for valid results?**

30 per variant is the minimum (Central Limit Theorem kicks in). 50+ is recommended when score variance is high or effect sizes are expected to be small. If your CI is very wide, increase `n_reps_per_variant` before drawing conclusions.

**Why low temperature for experiments?**

At high temperature, each LLM call introduces more sampling randomness. In an experiment, that randomness is noise — it inflates score variance, widens CIs, and requires more reps to reach significance. Run experiments at `temperature: 0.1–0.2` to isolate the prompt's effect. Use higher temperature only when testing whether a variant is robust to non-determinism.
